ThreadLocal

java
编程
发布日期:   2021-12-05
文章字数:   2.3k
阅读时长:   10 分
阅读次数:  

多线程访问同一个共享变量时，十分容易出现并发问题。为了保证线程安全，一般会采用同步的方式，同步措施一般需要加锁，除此之外还有一种较为轻量的方式，采用ThreadLocal方式防止共享变量读写冲突。

1.什么是ThreadLocal

ThreadLocal是通过线程隔离的方式防止任务在共享资源上产生冲突，线程本地存储是一种自动化机制，可以为使用相同变量的每个不同线程都创建不同的存储。
ThreadLocal为每个使用该变量的线程提供独立的变量副本，所以每一个线程都可以独立地改变自己的副本，而不会影响其它线程所对应的副本。

2.如何使用ThreadLocal

public class ThreadLocalTest {

  public static void main(String[] args) {

    ThreadLocal<Integer> threadLocal = new ThreadLocal<>();

    // thread 1
    Thread thread1 = new Thread(() -> {
      threadLocal.set(1);
      try {
        Thread.sleep(2000); // sleep 2s
        System.out.println(Thread.currentThread().getName() + ": " + threadLocal.get());
      } catch (InterruptedException e) {
        e.printStackTrace();
      }
    });

    // thread 2
    Thread thread2 = new Thread(() -> {
      threadLocal.set(2);
      try {
        Thread.sleep(1000); // sleep 1s
        System.out.println(Thread.currentThread().getName() + ": " + threadLocal.get());
      } catch (InterruptedException e) {
        e.printStackTrace();
      }
    });

    // start threads
    thread1.start();
    thread2.start();

  }

}

运行结果：

Thread-1: 2
Thread-0: 1

3.实现原理


Thread类中包含两个变量，threadLocals与inheritableThreadLocals。这两个变量是定制化的HashMap，在默认情况下都为null。只有当ThreadLocal第一次调用get或者set才会初始化。set的值最终是存放在Thread实例中，而不是ThreadLocal中，因此可以保证每个线程都有属于自己的变量副本。ThreadLocal就是以壳。

3.3 ThreadLocal类getMap、createMap方法

ThreadLocalMap getMap(Thread t) {
    return t.threadLocals;  // 直接返回线程的threadLocals变量
}

void createMap(Thread t, T firstValue) {
    t.threadLocals = new ThreadLocalMap(this, firstValue);  // key为ThreadLocal实例
}

3.4 ThreadLocal类set方法

public void set(T value) {
    Thread t = Thread.currentThread();  // 获取当前现场
    ThreadLocalMap map = getMap(t);  // 获取当前线程的threadLocals
    if (map != null)
        map.set(this, value);  // 将value存放至threadLocals中
    else
        createMap(t, value);  // 先初始化threadLocals,再设置value值
}

3.5 ThreadLocal类get方法

public T get() {
    Thread t = Thread.currentThread();  // 获取当前线程
    ThreadLocalMap map = getMap(t);  // 获取线程的threadLocals变量
    if (map != null) {
        ThreadLocalMap.Entry e = map.getEntry(this);  // 获取共享变量的所在的Entry
        if (e != null) {
            @SuppressWarnings("unchecked")
            T result = (T)e.value;  // 获取共享变量的值
            return result;
        }
    }
    return setInitialValue();  // 初始化threadLocals
}

private T setInitialValue() {
    T value = initialValue();  // 返回位空
    Thread t = Thread.currentThread();  
    ThreadLocalMap map = getMap(t);
    if (map != null)
        map.set(this, value);
    else
        createMap(t, value);  // 初始化
    return value;
}

protected T initialValue() {
    return null;  // 默认之间返回为空
}

3.6 ThreadLocal类remove方法

public void remove() {
    ThreadLocalMap m = getMap(Thread.currentThread());  // 获取线程的threadLocals变量
    if (m != null)
        m.remove(this);  // 调用ThreadLocalMap的remove方法
}

4.ThreadLocal使用场景

ThreadLocal的作用有保证线程安全和传递变量

4.1 保证线程安全

import java.text.ParseException;
import java.text.SimpleDateFormat;
import java.util.Date;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class DateUtil {

  private static final SimpleDateFormat sdf =
      new SimpleDateFormat("yyyy-MM-dd HH:mm:ss");

  public static Date parse(String dateStr) {
    Date date = null;
    try {
      date = sdf.parse(dateStr);
    } catch (ParseException e) {
      e.printStackTrace();
    }
    return date;
  }

  public static void main(String[] args) {
    ExecutorService service = Executors.newFixedThreadPool(10);

    for (int i = 0; i < 10; i++) {
      service.execute(()->{
        // parse线程不安全
        System.out.println(DateUtil.parse("2021-12-04 22:22:00"));
      });
    }
    service.shutdown();
  }
}

由于parse方法线程不安全，这个例子会出现报错。
可以采用ThreadLocal复制SimpleDateFormat实例。

import java.text.DateFormat;
import java.text.ParseException;
import java.text.SimpleDateFormat;
import java.util.Date;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class DateUtil {

  private static ThreadLocal<DateFormat> threadLocal = ThreadLocal.withInitial(
      ()-> new SimpleDateFormat("yyyy-MM-dd HH:mm:ss"));

  public static Date parse(String dateStr) {
    Date date = null;
    try {
      date = threadLocal.get().parse(dateStr);
    } catch (ParseException e) {
      e.printStackTrace();
    }
    return date;
  }

  public static void main(String[] args) {
    ExecutorService service = Executors.newFixedThreadPool(10);

    for (int i = 0; i < 10; i++) {
      service.execute(()->{
        System.out.println(DateUtil.parse("2021-12-04 22:22:00"));
      });
    }
    service.shutdown();
  }
}

看一下ThreadLocal.withInitial()源码

public static <S> ThreadLocal<S> withInitial(Supplier<? extends S> supplier) {
    return new SuppliedThreadLocal<>(supplier);
}

static final class SuppliedThreadLocal<T> extends ThreadLocal<T> {

    private final Supplier<? extends T> supplier;

    SuppliedThreadLocal(Supplier<? extends T> supplier) {
        this.supplier = Objects.requireNonNull(supplier);
    }

    // 只覆写了initialValue方法
    // 在ThreadLocal调用get方法，会调用setInitialValue方法
    // 在setInitialValue方法中，会先调用initialValue方法
    @Override
    protected T initialValue() {  
        return supplier.get();
    }
}

4.2 在线程级别传递变量

ThreadLocal 用作每个线程内需要独立保存信息，以便供其他方法更方便地获取该信息的场景。每个线程获取到的信息可能都是不一样的，前面执行的方法保存了信息后，后续方法可以通过ThreadLocal 直接获取到，避免了传参，类似于全局变量的概念。

5. 注意事项

5.1 内存泄露

static class ThreadLocalMap {

    /**
         * The entries in this hash map extend WeakReference, using
         * its main ref field as the key (which is always a
         * ThreadLocal object).  Note that null keys (i.e. entry.get()
         * == null) mean that the key is no longer referenced, so the
         * entry can be expunged from table.  Such entries are referred to
         * as "stale entries" in the code that follows.
         */
    static class Entry extends WeakReference<ThreadLocal<?>> {
        /** The value associated with this ThreadLocal. */
        Object value;

        Entry(ThreadLocal<?> k, Object v) {
            super(k);
            value = v;
        }
    }
}

由于Thread中包含变量ThreadLocalMap，因此ThreadLocalMap与Thread的生命周期是一样长，如果都没有手动删除对应key，都会导致内存泄漏。
但是使用弱引用可以多一层保障：弱引用ThreadLocal不会内存泄漏，对应的value在下一次ThreadLocalMap调用set(),get(),remove()的时候会被清除。
因此，ThreadLocal内存泄漏的根源是：由于ThreadLocalMap的生命周期跟Thread一样长，如果没有手动删除对应key就会导致内存泄漏，而不是因为弱引用。

// ThreadLocalMap类remove方法
private void remove(ThreadLocal<?> key) {
    Entry[] tab = table;
    int len = tab.length;
    int i = key.threadLocalHashCode & (len-1);
    for (Entry e = tab[i];
         e != null;
         e = tab[i = nextIndex(i, len)]) {
        if (e.get() == key) {
            e.clear();  // 清除key
            expungeStaleEntry(i); //清理key为null的元素
            return;
        }
    }
}

为什么ThreadLocal被设计成弱引用？
ThreadLocal的set方法最终是将值存在Entry类中，Entry继承WeakReference<ThreadLocal<?>>。key值为弱引用。
测试弱引用：

class ThreadLocalDemo {

  public static void main(String[] args) {
    ThreadLocal<Integer> integerThreadLocal = new ThreadLocal<>();

    Thread thread = Thread.currentThread();  // 观察threadLocals变量

    integerThreadLocal.set(3);  //　位置1 设置value为3 

    integerThreadLocal = null;  // 取消integerThreadLocal强引用

    System.gc();  // gc, 观察线程threadLocals变量

  }
}

执行完System.gc()：

可以看到垃圾回收之后，线程threadLocals变量中ThreadLocal对应的key值被回收了。
​

整个引用链条如下：
Thread -> ThreadLocal.ThreadLocalMap -> Entry[] -> Enrty -> key（threadLocal对象）和value
​

回归本质，ThreadLocalMap是用来存放对象的，在一次线程的执行栈中，存放数据后方便我们在任意的地方取得我们想要的值而不被其他线程干扰。ThreadLocalMap本身并没有为外界提供取出和存放数据的API，我们所能获得数据的方式只有通过ThreadLocal类提供的API来间接的从ThreadLocalMap取出数据，所以，当我们用不了key（ThreadLocal对象）的API也就无法从ThreadLocalMap里取出指定的数据。
在上面的例子中，integerThreadLocal对象被回收了，这些get和set方法也访问不到了，也就没法从ThreadLocalMap里取出数据了。没法利用API取出数据，那这个Entry对象还有用吗？所以最好的方法是在integerThreadLocal对象被回收后，系统自动回收对应的Entry对象，但是让Entry对象或其中的value对象做为弱引用都是非常不合理的。所以，让key（threadLocal对象）为弱引用，自动被垃圾回收，key就变为null了，下次，我们就可以通过Entry不为null，而key为null来判断该Entry对象该被清理掉了。
至于ThreadLocalMap为什么不给外界提供API来操作数据，我觉得是因为这个Map对于一个线程只有一份，任何地方都在用，为了提供更方便的API和为了我们不破换其他框架保存到里面的数据，所以才用ThreadLocal作为key和API来操作数据。
从另一个角度看，如果key值不为弱引用对象，那么ThreadLocal对象会被Thread对象强引用。一旦外部的ThreadLocal对象取消了强引用，但对象仍然不会被回收，从而造成了内存泄露。

5.2 脏数据

线程复用可能会产生脏数据。没有显性清理线程threadLocals变量数据，上一次使用线程时可能会存在值，再次使用时会读到上一次的值，出现脏数据。

5.3 解决办法

每次使用完ThreadLocal都调用它的remove()方法清除数据
